# ⚖️ 劳动法智能助手

基于**知识图谱 RAG（LightRAG）** + 大语言模型的劳动法咨询助手。能跨多条法规综合分析，针对真实劳动权益场景（被辞退、加班费、社保、三期保护等）给出带法条依据的解答；并支持上传**劳动合同 PDF / 工资条截图**做风险分析。

🌐 **在线体验**：http://47.242.111.247:8501

> ⚠️ 本项目仅供学习与技术演示，AI 生成内容不构成正式法律意见。

---

## ✨ 功能特点

- **知识图谱检索**：用 LightRAG 把劳动法规构建成实体-关系图谱，能顺着关系链跨多条法规综合回答（普通向量 RAG 做不到）
- **真实场景覆盖**：试用期、经济补偿(N/N+1/2N)、违法解除、加班费、社保、工伤、医疗期、三期女职工保护、竞业限制、劳动仲裁等高频场景（含补充法规语料 `labor_law_extra.txt`）
- **📄 劳动合同分析**：上传劳动合同 PDF，自动提取文字并扫描违法条款 / 风险条款 / 缺失条款（pdfplumber 解析）
- **🖼️ 图片识别分析**：上传工资条、合同截图、仲裁文书等图片，用多模态模型（Qwen2.5-VL）识别内容并从劳动法角度解读
- **💬 多轮对话 + 流式输出 + 历史持久化**：聊天式界面，支持多轮上下文追问、逐字流式输出，对话记录用 SQLite 本地存储
- **🛡️ 法条引用核验（真·防幻觉）**：不止靠免责声明——从知识库建立"真实存在的法条"索引，对回答里引用的每条法条自动核验，**编造的法条号当场标红**
- **📊 自动化评估**：内置 20 题评估集 + 评测脚本，可复现地给出关键点召回率、法条召回率、引用幻觉数

## 🏗️ 技术架构

支持两种运行形态：

**A. 单体完整版**（`law_app_v2.py`，单进程，部署最简单）
```
用户提问 / 上传文件
   │
   ▼
Streamlit（多 Tab：问答 / 合同分析 / 图片识别）
   ├── 智能问答 → LightRAG(hybrid) → DeepSeek-V3.1 → 法条引用核验
   ├── 合同分析 → pdfplumber → DeepSeek-V4-Flash 风险扫描
   └── 图片识别 → Qwen2.5-VL-72B 多模态
```

**B. 前后端分离版**（`law_app.py` + `law_api.py`，体现工程分层）
```
Streamlit 前端(law_app.py)  ──HTTP──▶  FastAPI 后端(law_api.py)
  界面/流式/多轮历史                      鉴权 + 限长 + RAG + 引用核验
```

**技术栈**：Python · LightRAG · Streamlit · FastAPI · DeepSeek-V3.1 / V4-Flash / Qwen2.5-VL（硅基流动 API）· Qwen3-Embedding · SQLite · pdfplumber

### 模型选型（按特长分工，实测得出）

| 环节 | 模型 | 为什么选它 |
|------|------|-----------|
| 建知识图谱 | DeepSeek-V3 | 实体抽取要稳定按格式输出；已建好存盘无需重建 |
| **知识图谱查询** | **DeepSeek-V3.1** | LightRAG 关键词抽取需要**结构化 response_format**，实测只有 V3/V3.1 支持，V4 系列会报 `response_format type unavailable` |
| 合同分析（纯生成） | DeepSeek-V4-Flash | 不走 LightRAG、不需要 response_format，V4-Flash 响应更快 |
| 图片识别 | Qwen2.5-VL-72B | 多模态视觉模型 |

> 这个分工是真实踩坑后定的：一开始想全用更快的 V4-Flash，结果发现它不支持 LightRAG 的结构化输出（见下方踩坑记录 4），于是改为"图谱查询走 V3.1、纯生成走 V4-Flash"各取所长。

#### ⚠️ 关于供应商：为什么用硅基流动，能不能换成 DeepSeek 官方？

本项目所有模型走**硅基流动（SiliconFlow）**一个 key。常问"能不能全换成 DeepSeek 官方 API？"——**不能完全换**，原因如下：

| 用途 | 当前模型 | DeepSeek 官方 API 有吗 |
|------|---------|----------------------|
| 建图 / 问答 / 合同分析 | DeepSeek-V3 / V3.1 / V4-Flash | ✅ 有（对话模型） |
| **向量化 Embedding** | Qwen3-Embedding | ❌ **没有** |
| **重排 Rerank** | Qwen3-Reranker-8B | ❌ **没有** |
| **图片识别（多模态）** | Qwen2.5-VL-72B | ❌ **没有** |

> **DeepSeek 官方 API（api.deepseek.com）只提供对话模型**，不提供 embedding / rerank / 视觉。
> 全换的后果：**RAG 检索（依赖 embedding）会瘫、重排失效、图片识别报废**。
>
> 硅基流动的价值正是"**聚合器**"——一个 key 同时给 DeepSeek 对话 + Qwen 的 embedding/rerank/VL，正好覆盖本项目所需。
>
> 若只想把**对话模型**换成 DeepSeek 官方（embedding 等仍留硅基流动）：改 `config.py` 的 base_url/key/模型名即可（都是 OpenAI 兼容接口），但模型名要改为官方命名 `deepseek-chat`（V3 系）/ `deepseek-reasoner`（推理），并把 config 改成支持对话与 embedding 用**不同 base_url/key**。属于"双供应商"方案，更复杂，没明确收益不建议。
>
> （DeepSeek 官方可能随时新增模型，决定前请到 platform.deepseek.com 再确认一次当前提供的接口类型。）

## 🔄 知识库数据管线：从"手敲静态库"到"对账式自动同步"

> 早期知识库是手写的 `labor_law.txt` 等静态语料——能跑，但不"真实"：数据怎么来、过期了怎么办，全靠人工。
> 这一版把它升级成**面向官方源的自动数据管线**，并解决了一个真实系统才会面对的难题：**失效数据的删除**。

### 1. 数据来源：只取官方权威源 + 全程溯源

法条的"真"不来自模型、不来自内容看起来对，而来自**出处权威性**。所以：

- `seed_laws.json`：钉死 14 部核心劳动法规的**官方 URL**（`gov.cn` / `npc.gov.cn` / `court.gov.cn`），定向抓取，不让爬虫自由发现，从源头杜绝脏数据。
- `crawl_npc.py`：抓正文 → **锚点裁剪**（定位"目录/第一条/颁布行"，丢掉页头导航和 CMS 碎片）→ 给每部法加 `【来源URL + 抓取日期】` 溯源头 → 内容哈希变更检测（变了才重抓）。
- **踩坑**：①人社部 `mohrss.gov.cn` 有反爬（返回 ~989 字节空壳）；②国家法律法规数据库 `flk.npc.gov.cn` 是 Vue SPA + PDF 式文档阅读器（正文被拆成碎片流式加载），极难干净抓取；③社媒爬虫（小红书/知乎类）源头不权威、绕登录爬个人数据有法律风险——**均否决**，最终选定 `gov.cn` 等纯 HTML 官方页。

### 2. 为什么不做"7×24 实时爬"——按数据变更频率设计

很多人第一反应是"持续实时爬才真实"。但**劳动法几年才修订一次**，实时爬是为高频数据（新闻/股价/社媒）设计的，用在这里是过度工程。
正确做法是**让同步频率匹配数据的真实变化节奏**：带变更检测的**定期增量同步**（每周/每月触发即可，用系统计划任务调度，与 Web 服务无关）。
> 工程判断力：先问"这数据多久变一次"，再决定同步策略——而不是无脑实时。

### 3. 对账式同步 `sync_graph.py`：增 / 改 / 删 / 去重

`add_docs.py` 只会**插入**；真实系统需要让图谱与权威源**对账（reconcile）一致**：

| 情形 | 处理 | 说明 |
|------|------|------|
| 新增 | 插入 | seed 里新增的法规 |
| 变更（修订） | 删旧版 + 插新版 | 按稳定 `doc_id` 原地替换 |
| **移除 / 废止** | **从图谱删除** | `add_docs` 做不到、却最关键的一环 |
| 重复 | 天然去重 | 每部法钉死稳定 doc_id，同一部永远一个文档 |

> **为什么"删除"最关键**：法律场景里，给用户答一条**已废止**的法条，比"查不到"更危险。
> 所以"实时/真实"的难点其实**不在爬取，而在失效数据的对账删除**——这才是玩具 RAG 和生产系统的分水岭。

### 4. 技术实现

- **稳定 doc_id**：`law-{md5(法名)}`，内容变了 ID 不变 → 更新即"删同 ID + 插同 ID"，干净原地替换（依赖 LightRAG `ainsert(ids=...)` 可指定文档 ID）。
- **删除**：LightRAG `adelete_by_doc_id`（≥1.4 支持）移除废止/修订的旧文档。
- **台账** `graph_manifest.json`：记录图谱中**受本管线管理**的法规 doc_id + 内容哈希（只动自己的 `law-*` 文档，不碰手写语料建的文档）。
- **`--dry-run`**：先预览对账计划（增/改/删几部）再执行，不动图谱、不烧 API。

```bash
python crawl_npc.py             # ① 抓取/更新本地缓存（官方源）
python sync_graph.py --dry-run  # ② 预览对账计划
python sync_graph.py            # ③ 执行同步：新增→插入、修订→替换、废止→删除
```

## 🧠 检索质量两层设计：重排（Reranker） vs 自动重检索（反思循环）

> 这两个常被混为一谈，其实在**不同层级**、互补叠加，不是二选一。

- **Reranker（重排）**：在**一次检索内部**，把召回的一堆候选**重新排序、挑最相关的**给模型。每次查询固定跑一次。
- **自动重检索（反思循环）**：模型**答完后**自检（引用够不够 / 有没有编法条），**不达标就重新再查一遍**。是套在最外层的循环，按答案质量触发 0~N 次。

```
[一次问答]
   召回(retrieve)
      ↓
   重排(reranker)        ← 在这一次检索内部，把对的内容排到前面
      ↓
   生成答案
      ↓
   自检(citation_check)  ← 引用够不够、有没有编造
      ↓ 不达标
   重检索 ───────────────┘  ← 反思循环：换个查询，再走一遍上面整条（含重排）
   （达标就输出）
```

| | Reranker | 自动重检索（反思循环） |
|---|---------|----------------------|
| 层级 | 单次检索**内部** | 跨多次检索的**外层循环** |
| 干什么 | 把**已召回**的候选重新排序 | 决定**要不要再查一遍** |
| 次数 | 固定 1 次 | 0~N 次（看答案质量） |
| 治什么病 | 对的内容**召回了但排太后**、被淹没 | 一次**没召回到 / 查偏了**，需换角度再查 |

**为什么两个都要**：Reranker 只能对**已经检索出来的那批**重排——如果第一遍压根没召回到对的 chunk，它再强也排不出不存在的东西；而自动重检索能**换查询、捞出不同的 chunk**，改变的是召回集本身。一个管"排得好"，一个管"查得够"，叠加使用。

> 本项目现状：Reranker 已落地（Qwen3-Reranker-8B，法条召回 60→70）；反思循环（`reflect_rag.py`）已实现核心（核验不达标→自动重检索），待接入主应用并用 eval 量化其增益。

## 🔬 案例：扩库为何让法条召回不升反降（eval 驱动归因）

> 一次"看似该变好、实测却变差"的真实复盘。比任何漂亮数字都能体现工程判断力。

**现象**：用官方源把知识库扩到 14 部全文（图谱 93→829 节点），本以为法条召回会涨，eval 却显示**从 60.7% 跌到 35.7%**（−25，远超 ±3~4 的生成噪声，是真回归）。

**归因（不靠猜，靠数据）**：写诊断脚本 `eval/diag_retrieval.py`，对每题用 `QueryParam(only_need_context=True)` dump 检索上下文，查期望条号在不在：

| 检查 | 结果 | 含义 |
|------|------|------|
| 条号（"第X条"）在上下文 | **100%** | 检索没丢东西 → **rerank/top-K 不是药**（差点用错） |
| 全称（"《法》第X条"）在上下文 | **仅 67%** | 问题在**格式**，不在检索 |

**两个洞叠加**导致召回低：

1. **格式洞**：爬来的全文正文是裸"第X条"，法名在另一个 chunk → 被切散 → 8 道新题的全称根本进不了上下文 → 注定 0%。
2. **生成洞**：即便全称在上下文（67%），模型也常**只说大意、不标条号** → 有料不引（35.7% < 67% 的差额即此）。

**两层修法 —— 对应"读时算 vs 写时冗余"的经典权衡**：

| 修法 | 层级 | 类比（DB/后端思维） | 治哪个洞 |
|------|------|--------------------|---------|
| **prompt 约束引用格式** | 读时（生成） | 读时计算 / 运行时 JOIN | 生成洞 |
| **chunk 注入法名**（每条文前补"《法名》"） | 写时（数据） | **反范式冗余**（denormalization） | 格式洞 |

> **关键认知**：冗余进 chunk 的必须是**人可读的"法名"**，不是机器 ID——模型不会拿 `law-abc123` 当法律依据引用，正如反范式要冗余 `user_name` 而非 `user_id`。两个洞**互补**：prompt 补不了格式洞（8 题上下文里连法名都没有），所以两个都得做。

**方法论价值**：eval + 诊断当场拦下了"加 rerank"的错误方向——检索已 100%，加重排纯属浪费。**RAG 调优必须证据驱动，不能凭直觉堆组件。**

> 状态：诊断已用数据实锤（检索 100% / 全称 67%）；修法（prompt 治生成洞 + chunk 冗余法名治格式洞）待实施并用 eval 验证。

## 🩺 法条召回攻坚：67.9% → 100%（eval 全程驱动）

> 上一案例诊断出"检索 100% 命中、生成却不引条号"。本轮把修法落地并用 eval 逐步验证，过程中暴露并解决了 4 类问题——其中一半其实是**评估自身的错**，不是模型的错。全程"改一处→重跑 eval"，凭数据决策。

### 问题与解决方案一览

| # | 问题 | 现象 | 根因 | 解决 |
|---|------|------|------|------|
| 1 | **评估"假漏引"** | 法条召回显示 0%，但答案明明引了条号 | `run_eval` 用裸子串匹配：模型写《**中华人民共和国**劳动合同法》第十九条，评估集写《劳动合同法》第十九条，中间隔了"中华人民共和国"→ 子串不连续 | 改用 `(法名归一化, 条号)` 集合比对，复用 `citation_check._normalize_law` 去前缀 → 召回 0%→67.9%（露出真实值） |
| 2 | **生成不引条号** | 检索到条号但答案不写出来（68% 封顶） | LightRAG 默认回答 prompt 不强制标注法条 | 注入 `CITATION_PROMPT`（查询 `user_prompt`）。eval A/B 实测：漏引题命中 **3/12 → 11/12**，召回 67.9%→96.4% |
| 2b | ⚠️ **负优化陷阱** | 泛泛要求"请标注法条"反而让原本有的条号**消失** | 措辞太泛 → 模型转去空谈、丢具体条号 | 精准措辞 + 防幻觉约束："只标检索内容中**确实出现**的条号，找不到就不标，**绝不臆造**"。这句既提引用率又压住编造 |
| 3 | **评估期望本身有错** | 题14/16/19 被误判扣分 | ①"两年"≠"二年"异形词；②等价法条只认一条（题16 模型引《劳动合同法》第三十条，eval 只认《劳动法》第五十条，二者法律等价）；③题19 塞了与本题无关的错误关键点"5天" | ①评分加"两↔二"归一；②eval 加 `cite_any`（等价法条任一命中即可）；③删错误关键点 |
| 4 | **知识库缺口致幻觉** | 题20 引《企业职工患病或非因工负伤医疗期规定》第三条 → 被核验判为幻觉 | 该法规**真实存在**（医疗期 3–24 个月的法定依据），但语料里只写了"依据《…规定》"、没按条号收录 → 核验索引查不到 | 经人社部官方源核实第三条原文，把语料补上正确条号 + 工龄对照表 → 核验实时收录，幻觉 1→0（无需重建图谱） |

### 关于"为什么不直接按法名+条号精确检索？"

一个很自然的问题：既然要的是法条，按"《X法》第Y条"精确匹配不是更准吗？答案是**二者互补，且精确检索不是本轮问题的解**：

- **本轮瓶颈不在检索**：诊断证明检索侧**条号 100% 命中**（24/24），问题在生成端引不引用。精确检索救不了"生成不写条号"。
- **用户提问绝大多数不带条号**："试用期多久""被违法辞退怎么赔"这类自然语言，**必须靠语义/图谱检索**把问题映射到法条；精确条号搜在这类 query 上无从下手。
- **精确检索的真正价值**是"已知条号查原文"场景，能保证返回**权威原文、杜绝转述出错**。**已落地为增强层**：`citation_check.build_article_index` 从 `npc_laws/` 14 部官方全文逐条切出 **711 条立法原文**，`law_api` 在引用核验时把已核实引用回填到 `sources` 字段——语义检索负责"问题→找到条号"，精确表负责"条号→权威原文"，各司其职、互为印证，而非互相替代。

### 最终结果（28 题全量）

| 指标 | 修评分前 | 真实基线 | 落地引用指令 | **A+B 修复后** |
|------|---------|---------|------------|--------------|
| 法条召回 | 假性偏低 | 67.9% | 96.4% | **100.0%** |
| 关键点召回 | 86.3% | 86.3% | 82.1% | **85.7%** |
| 引用幻觉 | 0 | 0 | 1 | **0** |

**方法论沉淀**：①每改一处必重跑 eval，CITATION_PROMPT 的负优化就是被 A/B 当场拦下的；②要分清"模型错"还是"评估错"——本轮 4 类问题里有一半是 eval 自身期望有误，盲目调模型只会南辕北辙。

## 🔧 工程实践 / 踩坑记录

> 这部分记录了开发中遇到的真实问题与解决思路。

1. **LightRAG 接国产 Embedding 的兼容问题**
   框架默认给 OpenAI 模型注入 `dimensions=1536` 参数做降维，但国产 Qwen Embedding（固定 1024 维）不支持，导致返回向量数量和维度错乱。通过逐层隔离测试（原生 API → openai SDK → 框架封装）定位根因，最终自实现 embedding 函数绕过，并关闭 base64 编码避免解码不兼容。

2. **Streamlit（同步）+ LightRAG（异步）的事件循环冲突**
   Streamlit 每次交互重跑脚本，`asyncio.run()` 反复新建事件循环，而 LightRAG 内部锁绑定到首次初始化的循环上，引发 "bound to a different event loop" 错误。解决方案：维护一个持久化事件循环，初始化与查询统一在该循环上执行。

3. **推理模型的 max_tokens 陷阱**
   引入 DeepSeek-V4-Flash 做合同分析时，回答时常空白或被截断。排查发现 V4 是**推理模型**，会先生成一大段思考（reasoning_content）再输出答案，原先较小的 `max_tokens` 被思考过程吃光。结论：用推理模型必须把 max_tokens 设足，否则正文没有预算。

4. **V4 不支持 LightRAG 的结构化 response_format**
   想把查询模型从 V3.1 升到更快的 V4-Flash，单测一个问题"通过"了，但跑评估集时大面积报错 `response_format type unavailable`。复盘发现：单测之所以"通过"是因为 LightRAG **缓存**了之前 V3.1 的关键词抽取结果，没真的调 V4。LightRAG 的关键词抽取依赖结构化 JSON 输出，实测只有 V3/V3.1 支持。**教训：验证要避开缓存、用未跑过的样本**——最终定为图谱查询用 V3.1、纯生成用 V4-Flash。

5. **从"能跑"到"工程化"的重构**
   初版把 `llm_func`/`embed_func` 在多个文件里复制粘贴、模型名硬编码、API 调用裸奔。重构为：`config.py` 集中配置、`llm_utils.py` 公共模块（含指数退避重试）、`law_api.py` 加接口鉴权与输入限长、新增法条引用核验与自动化评估。

## 🚀 本地运行

```bash
# 1. 安装依赖
pip install -r requirements.txt

# 2. 配置 Key：复制 .env.example 为 .env，填入硅基流动 Key（部署到云用 secrets，代码已兼容）
cp .env.example .env

# 3. 构建知识图谱（首次；之后复用 lightrag_store/）
python build_graph.py
python add_docs.py        # 可选：把补充法规 labor_law_extra.txt 增量并入图谱
```

**方式 A —— 单体完整版（推荐，一条命令）：**
```bash
streamlit run law_app_v2.py
```

**方式 B —— 前后端分离版（需起两个服务）：**
```bash
py -m uvicorn law_api:app --port 8000      # 终端1：后端
streamlit run law_app.py                   # 终端2：前端
```

浏览器访问 `http://localhost:8501`

## 📊 质量评估

法律问答正确性是生命线，因此内置可复现的自动化评估：

```bash
py eval/run_eval.py        # 跑全部 28 题
py eval/run_eval.py 5      # 只跑前 5 题（快速验证）
py eval/diag_retrieval.py  # 诊断：法条漏引是"检索没捞到"还是"生成没引"
```

评估三个指标：**关键点召回率**（标准答案要点覆盖度）、**法条召回率**（应引用法条的命中率）、**引用幻觉数**（编造的法条数，理想为 0）。

**当前实测结果**（28 题，知识图谱查询走 DeepSeek-V3.1，查询注入 `CITATION_PROMPT`）：

| 指标 | 数值 |
|------|------|
| 关键点平均召回率 | **85.7%** |
| 法条平均召回率 | **100.0%** ✅ |
| 引用幻觉总数 | **0** ✅ |

> 解读：**法条召回 100% + 0 幻觉**——完整攻坚过程见上文「法条召回攻坚 67.9%→100%」。关键不是堆模型，而是 eval 驱动地分清"检索问题/生成问题/评估问题/知识库缺口"四类，各个击破。关键点召回 85.7% 的剩余缺口多为"答对但少展开一两个要点"，是下一步打磨方向。
>
> 注：评估过程顶部出现过几次 `Connection Error`，被指数退避重试自动救回，28 题全部完成——侧面验证了重试机制有效。
